# 01 - 从聊天机器人到智能代理：范式转变的深层逻辑

> 理解 Agent 架构演进的第一步，是理解"对话"和"行动"之间的本质区别。

---

## 一、三代 AI 交互范式

AI 系统与人类的交互方式，经历了三个本质不同的阶段：

```
┌─────────────────────────────────────────────────────────────────────┐
│                   三代 AI 交互范式                                    │
│                                                                     │
│  第一代: 规则系统 (2010s)                                            │
│  ┌──────────────────────────────────────────────────┐               │
│  │ 用户输入 → 模式匹配 → 预设回复                     │               │
│  │ "今天天气怎么样" → match("天气") → callAPI()      │               │
│  │                                                  │               │
│  │ 特点: 确定性、有限状态、人工编写规则               │               │
│  │ 代表: Siri (早期)、小爱同学、客服机器人            │               │
│  └──────────────────────────────────────────────────┘               │
│                                                                     │
│  第二代: 对话式 AI (2022-2023)                                       │
│  ┌──────────────────────────────────────────────────┐               │
│  │ 用户输入 → LLM 理解 → 生成回复                     │               │
│  │ "帮我写一封邮件" → GPT-4 → "Dear ..."             │               │
│  │                                                  │               │
│  │ 特点: 涌现理解、自然语言、但只能"说"不能"做"       │               │
│  │ 代表: ChatGPT、Claude Chat、Gemini Chat           │               │
│  └──────────────────────────────────────────────────┘               │
│                                                                     │
│  第三代: 智能代理 Agent (2023-)                                      │
│  ┌──────────────────────────────────────────────────┐               │
│  │ 用户意图 → LLM 规划 → 工具执行 → 观察 → 继续...   │               │
│  │ "修复这个 bug" → 搜索代码 → 读取文件 → 编辑 →     │               │
│  │                  运行测试 → 确认修复 → 报告结果     │               │
│  │                                                  │               │
│  │ 特点: 自主规划、多步执行、与环境交互、闭环反馈     │               │
│  │ 代表: Claude Code、Cursor、Devin、Codex           │               │
│  └──────────────────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.1 从"回答"到"行动"——本质区别

**对话式 AI 的边界：**

```
用户: "帮我把 UserService 中的数据库调用抽取为 Repository 模式"

ChatGPT (对话模式):
  "好的，以下是重构的思路：
   1. 创建 UserRepository 接口...
   2. 实现 UserRepositoryImpl...
   3. 修改 UserService 注入 Repository...
   
   示例代码如下：
   ```typescript
   interface UserRepository { ... }
   ```"
   
   ← 输出了一段文本，但什么也没有真正改变
   ← 用户需要手动复制、粘贴、适配、调试
```

**Agent 的突破：**

```
用户: "帮我把 UserService 中的数据库调用抽取为 Repository 模式"

Claude Code (Agent 模式):
  [思考] 需要先了解当前代码结构...
  [行动] Grep: 搜索 UserService 相关文件
  [观察] 找到 src/services/UserService.ts, src/models/User.ts
  [行动] Read: 读取 UserService.ts
  [观察] 发现直接的 prisma 调用散落在 5 个方法中
  [行动] Write: 创建 src/repositories/UserRepository.ts
  [行动] Edit: 修改 UserService.ts，注入 Repository
  [行动] Bash: npx tsc --noEmit  (类型检查)
  [观察] 编译通过
  [行动] Bash: npm test  (运行测试)
  [观察] 所有测试通过
  [回复] "已完成重构。创建了 UserRepository 接口和实现，
          修改了 UserService 的 5 个方法..."
          
  ← 代码已经真正改变了
  ← 而且经过了验证
```

**关键差异总结：**

| 维度 | 对话式 AI | Agent |
|------|-----------|-------|
| 输出 | 文本（建议、代码片段） | 行动（实际执行操作） |
| 交互模式 | 一问一答 | 多步自主循环 |
| 环境感知 | 无（不知道你的代码库） | 有（能读取、搜索文件） |
| 反馈闭环 | 无（不知道建议是否有效） | 有（执行后验证结果） |
| 控制流 | 用户驱动每一步 | Agent 自主规划和执行 |
| 状态 | 无状态（对话窗口） | 有状态（文件系统、会话） |

## 二、Agent 定义的演进

"Agent" 这个词在不同时期有不同的含义，理解这个演进有助于避免概念混淆。

### 2.1 学术定义 (1990s-2010s)

来自分布式人工智能和多智能体系统研究：

```
Agent = 感知(Perception) + 推理(Reasoning) + 行动(Action)

          ┌──────────────┐
          │              │
  感知 ──→│   推理引擎    │──→ 行动
  (环境)  │  (规则/逻辑)  │   (改变环境)
          │              │
          └──────────────┘
               │   ↑
               └───┘
             内部状态

经典属性:
- 自主性 (Autonomy): 无需人类持续干预
- 反应性 (Reactivity): 能感知环境变化并响应
- 主动性 (Pro-activeness): 能主动追求目标
- 社会性 (Social ability): 能与其他 Agent 交互
```

### 2.2 LLM 时代的重新定义 (2023-)

当 LLM 成为推理引擎后，Agent 的定义发生了根本变化：

```
传统 Agent:  规则引擎 + 有限状态机 + 预定义行为
LLM Agent:   大语言模型 + 工具调用 + 自然语言规划

关键转变:
┌──────────────────────────────────────────────────────────────┐
│  Before LLM:                                                 │
│    推理能力弱 → 需要人类预编程所有逻辑                         │
│    理解能力弱 → 需要结构化输入（槽填充、意图分类）             │
│    生成能力弱 → 只能输出预设模板                               │
│                                                              │
│  After LLM:                                                  │
│    推理能力强 → LLM 自主规划执行步骤                          │
│    理解能力强 → 直接理解自然语言意图                           │
│    生成能力强 → 动态生成代码、文本、参数                       │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 当前工业界共识 (2025-2026)

经过两年的实践，行业对 "LLM Agent" 形成了相对清晰的共识：

```
LLM Agent = LLM (大脑)
           + Tools (手脚)
           + Memory (记忆)
           + Planning (规划)
           + Environment (环境)
```

**Anthropic 的定义（2025）：**
> "Agent 是一个使用 LLM 在循环中运行的系统，它能观察环境、做出决策、执行操作，
>  直到达成目标。关键区别在于 LLM 动态地指导程序的控制流。"

**OpenAI 的定义（2025）：**
> "Agent 是一个能独立代表你完成任务的系统。它结合了 LLM 的推理能力
>  和工具来感知、规划和行动于数字或物理世界。"

**共识要素：**

```
                    ┌─────────────────┐
                    │    用户意图       │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │   LLM (大脑)     │ ← 理解意图、规划步骤、
                    │                 │    做出决策、生成输出
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
     ┌──────────────┐ ┌──────────┐ ┌──────────────┐
     │  Tools (工具)  │ │ Memory   │ │ Environment  │
     │              │ │ (记忆)    │ │ (环境)        │
     │ - API 调用    │ │ - 短期   │ │ - 文件系统    │
     │ - 代码执行    │ │   (上下文)│ │ - 数据库     │
     │ - 数据查询    │ │ - 长期   │ │ - 浏览器     │
     │ - 文件操作    │ │   (向量库)│ │ - 终端       │
     └──────────────┘ └──────────┘ └──────────────┘
```

## 三、催生 Agent 的三大技术突破

Agent 架构不是凭空出现的，它依赖三个关键技术的成熟：

### 3.1 LLM 推理能力的飞跃

```
GPT-3 (2020):
  - 能生成流畅文本
  - 但逻辑推理弱，经常"胡说八道"
  - 无法可靠地遵循复杂指令
  → 不足以担任 Agent 的"大脑"

GPT-4 / Claude 3 (2023-2024):
  - 复杂推理能力大幅提升
  - 能理解和遵循多步骤指令
  - 能根据上下文做出合理决策
  - 能自我纠错（观察结果 → 调整策略）
  → 足以成为可靠的 Agent 决策引擎

GPT-4o / Claude 3.5 Sonnet / Claude 4 (2024-2026):
  - 推理成本大幅下降（Sonnet 比 Opus 便宜 5x，质量接近）
  - 多模态能力（图像理解、代码生成）
  - 更长的上下文窗口（128K-200K tokens）
  → Agent 从"技术演示"走向"生产可用"
```

### 3.2 Tool Use / Function Calling 的标准化

```
Before (2023.02 - ChatGPT Plugins):
  ┌──────────────────────────────────────────────────────────┐
  │ 工具调用方式: 在提示词中描述工具，让 LLM 输出特定格式文本，  │
  │ 然后用正则表达式解析。                                      │
  │                                                          │
  │ 问题:                                                     │
  │ - 格式不稳定（LLM 可能输出错误格式）                        │
  │ - 解析复杂（需要处理各种边界情况）                           │
  │ - 参数类型无保证（字符串 vs 数字 vs 布尔）                   │
  │ - 多工具调用难以可靠实现                                    │
  └──────────────────────────────────────────────────────────┘

After (2023.06 - OpenAI Function Calling):
  ┌──────────────────────────────────────────────────────────┐
  │ 工具调用方式: LLM 原生输出结构化 JSON，API 层保证格式正确。 │
  │                                                          │
  │ // API 请求中定义工具                                      │
  │ tools: [{                                                │
  │   type: "function",                                      │
  │   function: {                                            │
  │     name: "search_code",                                 │
  │     parameters: {                                        │
  │       type: "object",                                    │
  │       properties: {                                      │
  │         query: { type: "string" },                       │
  │         file_type: { type: "string", enum: [".ts",".js"]}│
  │       }                                                  │
  │     }                                                    │
  │   }                                                      │
  │ }]                                                       │
  │                                                          │
  │ // LLM 返回结构化调用                                      │
  │ tool_calls: [{                                           │
  │   function: {                                            │
  │     name: "search_code",                                 │
  │     arguments: '{"query":"UserRepository","file_type":".ts"}'│
  │   }                                                      │
  │ }]                                                       │
  │                                                          │
  │ 优势:                                                     │
  │ - 格式 100% 可靠（API 层保证）                             │
  │ - 参数类型安全（JSON Schema 校验）                          │
  │ - 支持并行多工具调用                                       │
  │ - 标准化接口，工具可复用                                    │
  └──────────────────────────────────────────────────────────┘
```

### 3.3 上下文窗口的扩大

```
上下文窗口大小直接决定了 Agent 的"工作记忆"容量:

模型            上下文窗口     约等于
──────          ──────────     ─────
GPT-3.5         4K tokens      ~3 页文本
GPT-4 (初始)    8K tokens      ~6 页文本
GPT-4-32K       32K tokens     ~25 页文本
Claude 2        100K tokens    ~75 页文本
GPT-4 Turbo     128K tokens    ~96 页文本
Claude 3        200K tokens    ~150 页文本
Gemini 1.5      1M tokens      ~750 页文本
Claude 4        200K tokens    ~150 页文本（质量更高）

对 Agent 的影响:
┌──────────────────────────────────────────────────────────────┐
│ 4K tokens:                                                   │
│   系统提示 + 几条消息就满了                                    │
│   → 只能做简单对话，不可能做 Agent                            │
│                                                              │
│ 32K tokens:                                                  │
│   系统提示 + 工具定义 + 约 10 轮工具调用                       │
│   → 可以做简单的 Agent，但容易"忘事"                          │
│                                                              │
│ 128K-200K tokens:                                            │
│   系统提示 + 工具定义 + 50+ 轮工具调用 + 多文件内容            │
│   → 可以处理复杂的多步骤任务                                  │
│   → Agent 终于有了足够的"工作记忆"                            │
│                                                              │
│ 1M tokens:                                                   │
│   几乎可以装下整个中型代码库                                   │
│   → "全局感知"成为可能                                        │
└──────────────────────────────────────────────────────────────┘
```

## 四、Agent 热潮的三波浪潮

### 4.1 第一波：全自主幻想 (2023.03 - 2023.09)

**标志事件：** AutoGPT 在 GitHub 上 3 天获得 10 万星

```
AutoGPT 的核心理念:
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  给 GPT-4 一个目标，让它完全自主地规划和执行               │
│                                                          │
│  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐  │
│  │ 设定   │───▶│ AI 规划│───▶│ AI 执行│───▶│ AI 评估│  │
│  │ 目标   │    │ 步骤   │    │ 操作   │    │ 结果   │  │
│  └────────┘    └────────┘    └────────┘    └────┬───┘  │
│                                                  │      │
│                     ◀────────────────────────────┘      │
│                         继续执行直到目标达成              │
│                                                          │
│  人类只需要: 1. 输入目标  2. 等待结果                     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**为什么失败了？**

```
理想:
  "创建一个电商网站" → AutoGPT → 3小时后 → 完整的电商网站

现实:
  "创建一个电商网站" → AutoGPT → 
    [第1步] 搜索"如何创建电商网站" 
    [第2步] 搜索"最好的电商框架"
    [第3步] 搜索"Next.js vs React"
    [第4步] 搜索"Next.js 教程"
    ... (无限循环搜索，从不开始写代码)
    [第50步] Token 耗尽，任务失败

核心问题:
1. 规划不可靠 — LLM 的长期规划能力不够，容易跑偏
2. 无纠错机制 — 错误会指数级放大（每步 5% 出错，20步后几乎必然失败）
3. 成本爆炸 — 每个循环都消耗 API tokens，无限循环 = 无限花钱
4. 缺乏人类监督 — 全自主 = 不可控
5. 工具能力弱 — 只有搜索和写文件，缺乏结构化工具
```

**这一波的价值：**
- 验证了 "LLM + Tools + Loop" 的基本架构方向是对的
- 暴露了全自主 Agent 的根本限制
- 催生了对 "有监督自主" 模式的探索

### 4.2 第二波：有监督的工具 Agent (2023.06 - 2024.12)

**标志事件：** OpenAI Function Calling + Assistants API

```
第二波的核心转变: "全自主" → "有监督的自主"

┌──────────────────────────────────────────────────────────┐
│                                                          │
│  不再追求让 AI 完全自主运行                               │
│  而是让 AI 成为人类的 "增强工具"                          │
│                                                          │
│  ┌─────────┐                      ┌─────────┐           │
│  │  人类    │ ── 下达指令 ───────▶│  Agent  │           │
│  │  (主导)  │                      │  (执行)  │           │
│  │         │◀── 请求确认/报告 ────│         │           │
│  │         │ ── 确认/调整 ───────▶│         │           │
│  │         │◀── 执行结果 ────────│         │           │
│  └─────────┘                      └─────────┘           │
│                                                          │
│  人类角色: 监督者 + 决策者 + 纠错者                       │
│  Agent 角色: 执行者 + 建议者 + 信息整合者                 │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**代表产品与架构：**

```
OpenAI Assistants API (2023.10):
├── 有状态 Thread（会话持久化）
├── 内置工具:
│   ├── Code Interpreter（沙箱代码执行）
│   ├── File Search（RAG 检索）
│   └── Function Calling（自定义函数）
├── 异步运行模型（Run → Steps → 轮询/Streaming）
└── 重要限制: 云端运行，不能操作本地环境

Claude Code (2024.06):
├── 本地 Agentic Loop
├── 内置开发工具（Read, Write, Edit, Bash, Grep, Glob）
├── 多层权限模型（只读自动允许，写入需确认）
├── CLAUDE.md 项目指令（让 Agent 理解项目约定）
└── 关键创新: 本地文件系统操作 + 权限安全模型

Cursor / Windsurf (2024):
├── IDE 内嵌 Agent
├── 代码编辑 Diff 模式
├── 多文件协同编辑
├── 上下文自动收集（打开的文件、光标位置、lint 错误）
└── 关键创新: 深度融入开发者工作流
```

### 4.3 第三波：Multi-Agent 与 Skills 组合 (2025-)

**标志事件：** OpenAI Agents SDK + Claude Code Skills + A2A Protocol

```
第三波的核心理念:

"一个 Agent 做不好所有事，不如让多个专精的 Agent 协作"

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  单 Agent 的瓶颈:                                            │
│  ┌────────────────────────────────────────────────────┐     │
│  │                     单一 Agent                      │     │
│  │                                                    │     │
│  │  1. 上下文窗口有限 → 不能同时处理太多信息            │     │
│  │  2. 能力扩展受限 → 一个 system prompt 塞不下太多指令  │     │
│  │  3. 专精矛盾 → 对所有领域都"略知一二"，不够精通      │     │
│  │  4. 错误累积 → 长任务链中错误概率叠加                 │     │
│  │  5. 延迟瓶颈 → 串行执行，无法并行                    │     │
│  └────────────────────────────────────────────────────┘     │
│                          │                                   │
│                          ▼                                   │
│  Multi-Agent 的解决:                                         │
│  ┌────────────────────────────────────────────────────┐     │
│  │              Orchestrator Agent                      │     │
│  │         (理解意图、分解任务、协调执行)                 │     │
│  │                                                    │     │
│  │    ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐        │     │
│  │    │代码   │  │测试   │  │文档   │  │安全   │        │     │
│  │    │Agent  │  │Agent  │  │Agent  │  │Agent  │        │     │
│  │    │      │  │      │  │      │  │      │        │     │
│  │    │专精于 │  │专精于 │  │专精于 │  │专精于 │        │     │
│  │    │代码编写│  │测试用例│  │文档撰写│  │安全审计│       │     │
│  │    └──────┘  └──────┘  └──────┘  └──────┘        │     │
│  │                                                    │     │
│  │  优势:                                              │     │
│  │  1. 每个 Agent 的上下文只包含相关信息                 │     │
│  │  2. 每个 Agent 有专精的 system prompt 和工具集        │     │
│  │  3. 可并行执行独立任务                               │     │
│  │  4. 可独立验证和重试                                 │     │
│  └────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────┘
```

## 五、一个关键抽象：Agent 的四层架构

经过三波浪潮的洗礼，Agent 架构逐渐收敛到一个四层抽象模型：

```
┌──────────────────────────────────────────────────────────────┐
│                    Agent 四层架构模型                          │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ Layer 4: Orchestration (编排层)                         │  │
│  │                                                        │  │
│  │ 负责多 Agent 的协调、任务分解、结果聚合                   │  │
│  │ 模式: Router / Pipeline / Graph / Swarm                │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ Layer 3: Skills (能力层)                                │  │
│  │                                                        │  │
│  │ 可复用的能力单元，封装了 system prompt + tools + logic   │  │
│  │ 例如: "代码审查 Skill"、"数据分析 Skill"                │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ Layer 2: Tools (工具层)                                 │  │
│  │                                                        │  │
│  │ 原子级操作，与外部世界交互的接口                         │  │
│  │ 例如: read_file, search_code, run_command, http_request │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ Layer 1: LLM Core (推理层)                              │  │
│  │                                                        │  │
│  │ 大语言模型本身，提供理解、推理、生成能力                  │  │
│  │ 通过 Agentic Loop 驱动整个系统运转                       │  │
│  └────────────────────────────────────────────────────────┘  │
│                                                              │
│  各层之间的关系:                                              │
│  - Layer 1 是基础，决定了 Agent 的"智力上限"                 │
│  - Layer 2 扩展了 Agent 的"感知和行动范围"                   │
│  - Layer 3 提供了"领域专业知识和最佳实践"                    │
│  - Layer 4 实现了"分工协作和规模化"                          │
└──────────────────────────────────────────────────────────────┘
```

这四层架构也对应了本系列文档的主要内容——接下来的每篇文档将深入探讨其中一个层面。

## 六、Agent vs. Workflow: 一个重要的区分

Anthropic 在 2025 年的一篇重要博客中提出了一个关键区分：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  Agentic System (Agent 系统) 包含两种模式:                       │
│                                                                 │
│  1. Workflow (工作流):                                           │
│     LLM 和工具按照 预定义的代码路径 编排                          │
│     人类预先设计了执行流程，LLM 只是流程中的一个节点               │
│                                                                 │
│     例如:                                                        │
│     用户输入 → 分类器(LLM) → if 技术问题:                        │
│                                   查知识库 → 生成回答(LLM)       │
│                                else:                            │
│                                   转人工                        │
│                                                                 │
│  2. Agent (代理):                                                │
│     LLM 动态地决定控制流 —— 调什么工具、什么顺序、执行多少步       │
│     人类只提供目标和工具，执行策略由 LLM 自主决定                  │
│                                                                 │
│     例如:                                                        │
│     用户意图 → LLM → [自主决定] → Tool A → [观察] →             │
│                      [自主决定] → Tool C → [观察] →             │
│                      [自主决定] → 输出结果                       │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    确定性                                 │   │
│  │                       ↑                                  │   │
│  │   纯代码          Workflow          Agent          全自主 │   │
│  │  (if/else)     (LLM 嵌入流程)   (LLM 驱动流程)    (AutoGPT)│   │
│  │     ●──────────●───────────────●───────────────●         │   │
│  │                       ↓                                  │   │
│  │                    灵活性                                 │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Anthropic 的建议:                                               │
│  "从最简单的方案开始，只在需要更多灵活性时才增加复杂度"            │
│  → 很多场景用 Workflow 就够了，不需要真正的 Agent                 │
│  → Agent 适合开放性、探索性的任务（如 coding、research）          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**实际框架的定位：**

```
纯 Workflow                                      纯 Agent
    ←──────────────────────────────────────────────→
    
    Prefect       LangGraph     CrewAI     Claude Code    AutoGPT
    Airflow       (图状态机)    (角色扮演)  (Agentic Loop)  (全自主)
    (DAG 编排)                                             
    
    确定性高                                        灵活性高
    可预测                                          不可预测
    适合批处理                                      适合探索性任务
```

这个区分很重要：**并非所有使用 LLM 的系统都是 Agent，也不是所有场景都需要 Agent。** 理解何时用 Workflow、何时用 Agent，是架构设计的关键决策。

## 总结

从聊天机器人到 Agent 的演进，本质上是三个维度的跃迁：

1. **从"回答"到"行动"**：AI 不再只是给建议，而是真正执行操作
2. **从"被动"到"主动"**：AI 不再等待每一步指令，而是自主规划和执行
3. **从"孤立"到"连接"**：AI 不再只在对话窗口中运行，而是连接到真实的工具和环境

```
         回答                行动
          ↑                   ↑
          │     ┌─────┐      │
    被动 ─┤     │Agent│      ├─ 主动
          │     └─────┘      │
          │                   │
         孤立               连接

ChatGPT: 回答 + 被动 + 孤立 (左下角)
Agent:   行动 + 主动 + 连接 (右上角)
```

接下来，我们将深入每个层面的技术细节，看看 Agent 架构是如何一步步走到今天的。
