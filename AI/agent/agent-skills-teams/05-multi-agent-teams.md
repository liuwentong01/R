# 05 - Multi-Agent Teams 架构

> 当一个 Agent 不够用时，让多个专精的 Agent 组成团队协作。本文深入分析多 Agent 系统的设计模式、通信机制和协作策略。

---

## 一、为什么需要多 Agent

### 1.1 单 Agent 的天花板

```
单 Agent 遇到的五大瓶颈:

1. Context Window 瓶颈
   ┌──────────────────────────────────────────────────────┐
   │ 一个复杂的全栈开发任务需要:                            │
   │ - 前端框架文档 (10K tokens)                          │
   │ - 后端 API 规范 (8K tokens)                          │
   │ - 数据库 Schema (5K tokens)                          │
   │ - 安全最佳实践 (4K tokens)                           │
   │ - 测试规范 (3K tokens)                               │
   │ - 项目编码规范 (3K tokens)                           │
   │ = 总计 33K tokens 仅 system prompt                   │
   │ + 工具定义 15K + 对话历史 100K+ = 溢出               │
   └──────────────────────────────────────────────────────┘

2. 专精悖论
   "样样通，样样松" — 一个 agent 要同时精通前端、后端、
   数据库、安全、测试、DevOps，instructions 必然泛泛而谈

3. 错误传播
   20 步任务，每步 95% 成功率 → 总成功率 = 0.95^20 ≈ 36%
   但如果分给 4 个 Agent 各做 5 步 → 每个 0.95^5 ≈ 77%
   且失败的 Agent 可以独立重试，不影响其他 Agent

4. 延迟瓶颈
   串行执行 20 个工具调用需要 20 次 LLM 调用
   但如果 10 个独立步骤可以并行 → 总延迟减半

5. 可维护性
   一个超长的 system prompt → 难以维护、测试、调试
   多个专精 Agent → 每个都可以独立开发、测试、迭代
```

### 1.2 多 Agent 的核心价值

```
Multi-Agent 本质上是 "分治法 (Divide and Conquer)" 在 AI 领域的应用:

  复杂任务 → 分解为子任务 → 分配给专精 Agent → 独立执行 → 聚合结果

┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  单 Agent:                                                       │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    Super Agent                            │   │
│  │  知道一点前端、一点后端、一点测试、一点安全...             │   │
│  │  200K context 全塞满                                      │   │
│  │  串行处理所有任务                                         │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  Multi-Agent:                                                    │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │ Frontend     │  │ Backend      │  │ QA           │          │
│  │ Agent        │  │ Agent        │  │ Agent        │          │
│  │              │  │              │  │              │          │
│  │ 精通 React   │  │ 精通 Node.js │  │ 精通测试     │          │
│  │ 50K context  │  │ 50K context  │  │ 50K context  │          │
│  │ 专用工具集   │  │ 专用工具集   │  │ 专用工具集   │          │
│  └──────────────┘  └──────────────┘  └──────────────┘          │
│                                                                  │
│  对比:                                                           │
│  - 上下文: 200K (拥挤) vs 3×50K (宽裕)                          │
│  - 专精度: 泛泛而谈 vs 领域专家                                  │
│  - 延迟: 全串行 vs 可并行                                       │
│  - 可维护: 一个巨大 prompt vs 三个小 prompt                      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 二、Multi-Agent 通信模式

Agent 之间如何传递信息，是多 Agent 系统设计的核心问题。

### 2.1 Handoff 模式 (接力棒)

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Handoff 模式                                       │
│                                                                      │
│  核心思想: 一个 Agent 完成后，将控制权"交接"给下一个 Agent            │
│  类比: 接力赛跑 — 跑完自己的赛道，把接力棒传给下一个选手              │
│                                                                      │
│  ┌────────┐  handoff   ┌────────┐  handoff   ┌────────┐            │
│  │Agent A │ ─────────▶ │Agent B │ ─────────▶ │Agent C │            │
│  │(规划)  │  传递:      │(编码)  │  传递:      │(测试)  │            │
│  └────────┘  - 上下文   └────────┘  - 上下文   └────────┘            │
│              - 任务描述              - 代码变更                       │
│              - 约束条件              - 需要测试的内容                  │
│                                                                      │
│  OpenAI Agents SDK 的 Handoff 实现:                                  │
│                                                                      │
│  triage_agent = Agent(                                               │
│      name="Triage",                                                  │
│      instructions="根据用户请求类型转交给合适的专家 Agent",            │
│      handoffs=[frontend_agent, backend_agent, devops_agent]          │
│  )                                                                   │
│                                                                      │
│  # 当 triage_agent 决定 handoff 时:                                  │
│  # 1. 当前 Agent 停止执行                                            │
│  # 2. 对话历史传递给目标 Agent                                       │
│  # 3. 目标 Agent 从接收到的上下文继续执行                             │
│  # 4. 目标 Agent 也可以 handoff 回来或给其他 Agent                    │
│                                                                      │
│  特点:                                                               │
│  ✅ 简单直观 — 同一时间只有一个 Agent 活跃                           │
│  ✅ 上下文连贯 — 对话历史完整传递                                    │
│  ❌ 串行执行 — 无法并行                                              │
│  ❌ 上下文膨胀 — 历史越来越长                                        │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 Orchestrator 模式 (指挥家)

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Orchestrator 模式                                   │
│                                                                      │
│  核心思想: 一个中心 Agent (Orchestrator) 协调多个 Worker Agent         │
│  类比: 交响乐指挥家 — 不自己演奏，但协调所有乐手                      │
│                                                                      │
│                    ┌────────────────┐                                │
│                    │  Orchestrator  │                                │
│                    │  (指挥家)       │                                │
│                    │                │                                │
│                    │  职责:         │                                │
│                    │  - 理解任务    │                                │
│                    │  - 分解子任务  │                                │
│                    │  - 分配 Agent  │                                │
│                    │  - 聚合结果    │                                │
│                    │  - 处理冲突    │                                │
│                    └───┬───┬───┬───┘                                │
│                        │   │   │                                    │
│              ┌─────────┘   │   └─────────┐                          │
│              │             │             │                          │
│              ▼             ▼             ▼                          │
│        ┌──────────┐ ┌──────────┐ ┌──────────┐                     │
│        │ Worker A │ │ Worker B │ │ Worker C │                     │
│        │ (前端)   │ │ (后端)   │ │ (数据库) │                     │
│        └──────────┘ └──────────┘ └──────────┘                     │
│                                                                      │
│  Claude Code 的 SubAgent 就是这个模式:                                │
│                                                                      │
│  主 Agent (Orchestrator):                                            │
│    "我需要同时做三件事:                                               │
│     1. 搜索所有使用旧 API 的文件                                     │
│     2. 读取新 API 的文档                                             │
│     3. 检查测试覆盖率"                                               │
│                                                                      │
│    → 调用 3 个 Task 工具，启动 3 个 SubAgent 并行执行                 │
│    → 等待所有结果返回                                                │
│    → 综合分析后执行迁移                                              │
│                                                                      │
│  特点:                                                               │
│  ✅ 可并行 — Worker 之间独立执行                                     │
│  ✅ 灵活 — Orchestrator 动态决定分配策略                              │
│  ✅ 可扩展 — 轻松增加新的 Worker 类型                                │
│  ❌ 单点瓶颈 — Orchestrator 本身成为瓶颈                             │
│  ❌ 通信开销 — 每次交互都经过 Orchestrator                           │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.3 Peer-to-Peer 模式 (同僚协作)

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Peer-to-Peer 模式                                   │
│                                                                      │
│  核心思想: Agent 之间直接通信，没有中心协调者                          │
│  类比: 小组讨论 — 成员平等交流，协商达成共识                          │
│                                                                      │
│        ┌──────────┐                                                 │
│        │ Agent A  │◄────────────┐                                  │
│        │ (设计师) │             │                                  │
│        └────┬─────┘             │                                  │
│             │                   │                                  │
│             ▼                   │                                  │
│        ┌──────────┐       ┌──────────┐                             │
│        │ Agent B  │◄─────▶│ Agent C  │                             │
│        │ (开发者) │       │ (测试员) │                             │
│        └──────────┘       └──────────┘                             │
│                                                                      │
│  CrewAI 的 Crew 模式:                                                │
│                                                                      │
│  crew = Crew(                                                        │
│      agents=[designer, developer, tester],                           │
│      tasks=[                                                         │
│          Task(description="设计 API", agent=designer),               │
│          Task(description="实现 API", agent=developer,               │
│               context=[designer_task]),  # 依赖设计结果               │
│          Task(description="测试 API", agent=tester,                  │
│               context=[developer_task])  # 依赖实现结果               │
│      ],                                                              │
│      process=Process.sequential  # 或 hierarchical                   │
│  )                                                                   │
│                                                                      │
│  特点:                                                               │
│  ✅ 去中心化 — 没有单点故障                                          │
│  ✅ 自然协作 — 类似人类团队                                          │
│  ❌ 协调复杂 — 可能出现死锁、冲突                                    │
│  ❌ 不确定性高 — 难以预测执行路径                                    │
│  ❌ 调试困难 — 多 Agent 交互难以追踪                                 │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.4 Broadcast 模式 (广播)

```
┌──────────────────────────────────────────────────────────────────────┐
│                    Broadcast 模式                                     │
│                                                                      │
│  核心思想: 一个 Agent 的输出广播给所有其他 Agent                      │
│  类比: 会议演讲 — 一人发言，所有人听到                                │

│           │           │           │           │                     │

│     ┌──────────┐┌──────────┐┌──────────┐┌──────────┐              │
│     │Security  ││Perf      ││Style     ││Logic     │              │

│     └────┬─────┘└────┬─────┘└────┬─────┘└────┬─────┘              │

└──────────────────────────────────────────────────────────────────────┘
```

## 四、Agent 间的状态共享

### 4.1 共享状态的挑战

```
多 Agent 系统中最难的问题之一: 如何共享状态？

问题场景:
  Agent A 修改了 src/auth.ts
  Agent B 也要修改 src/auth.ts
  → 冲突！

Agent A 发现了一个 bug 影响 Agent B 的任务
  → Agent B 需要知道这个信息
  → 但 Agent B 正在独立运行，有自己的上下文

Agent C 完成了数据库 migration
  → Agent A 和 Agent B 都需要知道表结构变了
```

### 4.2 状态共享策略

```
策略 1: 共享文件系统 (最常见)
┌──────────────────────────────────────────────────────────────┐
│ 所有 Agent 操作同一个文件系统                                 │
│                                                              │
│ Agent A: write_file("src/auth.ts", ...)                     │
│ Agent B: read_file("src/auth.ts") → 看到 A 的修改           │
│                                                              │
│ Claude Code 的做法:                                          │
│ - 主 Agent 和 SubAgent 共享同一个工作目录                    │
│ - SubAgent 的文件修改对主 Agent 立即可见                     │
│ - 本质上通过文件系统实现隐式状态共享                         │
│                                                              │
│ 问题: 没有并发控制 → 可能出现写写冲突                        │
│ 缓解: 通常 Orchestrator 会避免分配冲突的文件给不同 Agent     │
└──────────────────────────────────────────────────────────────┘

策略 2: 消息传递 (Handoff 中的上下文传递)
┌──────────────────────────────────────────────────────────────┐
│ Agent 间通过消息传递状态信息                                  │
│                                                              │
│ Handoff 消息包含:                                            │
│ - 完整对话历史 (或摘要)                                      │
│ - 当前任务状态                                               │
│ - 关键发现和决策                                             │
│                                                              │
│ OpenAI Agents SDK:                                           │
│ handoff 时自动传递完整 message history                       │
│ → 下一个 Agent 看到所有之前的对话                            │
│                                                              │
│ 问题: 历史太长 → context 溢出                                │
│ 缓解: 传递前做摘要压缩                                       │
└──────────────────────────────────────────────────────────────┘

策略 3: 共享内存/黑板 (Blackboard Pattern)
┌──────────────────────────────────────────────────────────────┐
│ 所有 Agent 读写一个共享的 "黑板"                              │
│                                                              │
│             ┌────────────────────────┐                       │
│             │      Blackboard        │                       │
│             │                        │                       │
│             │ requirements: [...]    │                       │
│             │ architecture: {...}    │                       │
│             │ code_changes: [...]    │                       │
│             │ test_results: {...}    │                       │
│             │ issues_found: [...]    │                       │
│             └─────────┬──────────────┘                       │
│               ▲  ▲  ▲ │ ▼  ▼  ▼                             │
│             ┌─┘  │  │ │ │  │  └─┐                           │
│             │    │  └─┼─┘  │    │                           │
│           ┌─┴─┐┌─┴─┐ │ ┌──┴─┐┌─┴──┐                       │
│           │ A ││ B │ │ │ C  ││ D  │                       │
│           └───┘└───┘ │ └────┘└────┘                       │
│                       │                                     │
│  LangGraph 实现:       │                                     │
│  State 对象就是黑板    │                                     │
│  每个 Node (Agent) 读写 State                               │
│  Graph 引擎管理并发和状态一致性                               │
└──────────────────────────────────────────────────────────────┘

策略 4: 事件总线 (Event Bus)
┌──────────────────────────────────────────────────────────────┐
│ Agent 通过发布/订阅事件进行通信                               │
│                                                              │
│ Agent A: emit("file_changed", { file: "src/auth.ts" })     │
│ Agent B: on("file_changed", (e) => { /* 更新我的上下文 */ }) │
│                                                              │
│ OpenClaw 的做法:                                             │
│ - 25 个生命周期钩子 = 事件系统                               │
│ - Plugin 间通过钩子通信                                      │
│ - 例: before_agent_run → after_agent_run → message_sent     │
└──────────────────────────────────────────────────────────────┘
```

## 五、Multi-Agent 的关键设计决策

### 5.1 同步 vs 异步

```
同步 Multi-Agent:
  Orchestrator 发出任务 → 等待所有 Worker 完成 → 聚合结果
  
  ┌───┐  task  ┌───┐
  │ O │──────▶│ A │──── 30s ────▶ result A
  │ r │  task  ├───┤                  │
  │ c │──────▶│ B │──── 45s ────▶ result B  → 聚合 → 继续
  │ h │  task  ├───┤                  │
  │   │──────▶│ C │──── 20s ────▶ result C
  └───┘       └───┘
  
  总耗时: max(30, 45, 20) = 45s (并行) 或 95s (串行)
  
  适合: 任务有明确的同步点（如 "所有审查完成后再合并"）

异步 Multi-Agent:
  Agent 独立运行，通过事件/消息松散协作
  
  Agent A: 发现 bug → 发事件 → 不等待，继续自己的工作
  Agent B: 收到事件 → 在合适的时候处理
  
  适合: 长时间运行的后台任务（如 Cursor Background Agents）
  
  Codex (OpenAI) 的做法:
  - 用户提交 Task → 云端 Agent 异步执行
  - Agent 在沙箱中独立运行（可能几分钟到几小时）
  - 完成后通知用户，创建 PR
  - 多个 Task 可以并行执行
```

## 七、Multi-Agent 的反模式

```
┌──────────────────────────────────────────────────────────────────┐
│                    Multi-Agent 反模式                              │
│                                                                  │
│  1. 过度拆分 (Over-decomposition)                                │
│     ❌ 10 个 Agent 做一个简单 CRUD → 通信开销 >> 实际工作         │
│     ✅ 简单任务用单 Agent + 多 Tool 就够了                        │
│     规则: 只有当任务确实需要不同专业知识或需要并行时才拆分         │
│                                                                  │
│  2. Agent 话痨 (Chatty Agents)                                   │
│     ❌ Agent 之间频繁来回对话，大量 token 消耗在"沟通"上          │
│     ✅ 明确定义每次 handoff 传递的信息，避免不必要的来回          │
│                                                                  │
│  3. 缺乏终止条件                                                 │
│     ❌ Agent A handoff 给 B, B handoff 回 A, 无限循环            │
│     ✅ 设置 max_turns，检测循环 handoff                          │
│                                                                  │
│  4. 全用强模型                                                    │
│     ❌ 每个 Agent 都用 Claude 4 Opus → 成本爆炸                  │
│     ✅ 按角色需求选模型（Router 用 mini，Coder 用 Sonnet）        │
│                                                                  │
│  5. 忽略错误传播                                                  │
│     ❌ Agent A 的错误结果传给 Agent B，B 基于错误继续工作          │
│     ✅ 每个 Agent 的输出经过验证，错误及时发现和纠正              │
│                                                                  │
│  Anthropic 的建议:                                                │
│  "从单 Agent + Prompt Chaining 开始。                             │
│   只有在确认单 Agent 方案不够时，才引入 Multi-Agent。              │
│   大部分场景不需要 Multi-Agent。"                                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 总结

```
Multi-Agent Teams 架构的核心抽象:

  ┌─────────────────────────────────────────────────────────┐
  │                                                         │
  │  Roles (角色)     — 每个 Agent 的专精领域和行为规范      │
  │  Communication    — Agent 之间如何传递信息               │
  │  Coordination     — 谁来决定谁做什么、什么时候做         │
  │  Shared State     — Agent 如何共享和同步状态             │
  │  Termination      — 什么时候整体任务算完成               │
  │                                                         │
  └─────────────────────────────────────────────────────────┘

通信模式选择:
  Handoff:       适合串行流水线（分诊→处理→审查）
  Orchestrator:  适合任务分解和并行（最常用）
  Peer-to-Peer:  适合复杂协商（但实现难度高）
  Broadcast:     适合多角度分析（审查、投票）

核心经验:
  1. 不要为了用 Multi-Agent 而用 — 简单任务单 Agent 更好
  2. 明确每个 Agent 的角色边界 — 避免职责重叠
  3. 控制通信开销 — Agent 间传递最少必要信息
  4. 设置安全阀 — max_turns, 错误阈值, 超时
  5. 混合模型策略 — 按角色选择性价比最优的模型
```
